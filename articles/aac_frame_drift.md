# 動画を分割したら4.5秒ズレた（と思った）話

## 「先生、早すぎます」

AIエージェント8体で切り抜き動画を作っている。

ある日、ゲーム実況の切り抜きショートを作っていた。106分の長尺動画から「先生が問題を解読するシーン」を切り出す作業だ。

字幕データから「正解を見ますか」という発言のタイムスタンプを取得。**30分17秒748ミリ秒**。ミリ秒精度。完璧だ。

ffmpegでその時刻から切り出して、YouTubeにアップ。殿（人間の上司）に確認してもらう。

**「頭に4秒くらい余計なの入ってるよ」**

えっ。

---

## 犯人探し

うちのパイプラインはこうなっている。

```
元動画（106分）
  ↓ ffmpegで10分チャンクに分割
チャンク0〜10（各10分）
  ↓ Demucs（BGM除去、ボーカルだけ抽出）
ボーカル音声ファイル
  ↓ AssemblyAI（音声→テキスト変換）
単語ごとのタイムスタンプ付きJSON
```

このJSONから「正解」という単語のタイムスタンプを取って、元動画をffmpegで切り出す。シンプル。なのに4.5秒ズレる。

犯人は誰だ。

---

## 容疑者1: Demucs（BGM除去AI）

「BGMを除去する過程で音声がズレたのでは？」

各チャンクの入力尺と出力尺を比較。

```
chunk_000: 入力600.015秒 → 出力599.915秒（差 0.100秒）
chunk_001: 入力600.000秒 → 出力599.900秒（差 0.100秒）
...
chunk_010: 入力363.833秒 → 出力363.826秒（差 0.007秒）
```

各チャンクで0.1秒短くなっている！11チャンクで累積すると…**0.985秒**。

1秒のズレは説明できる。でも4.5秒には全然足りない。

**Demucs: 容疑は薄い（でも0.1秒/チャンクの損失は気になる）**

---

## 容疑者2: AssemblyAI（音声認識API）

「長時間の音声を処理すると、後半のタイムスタンプがドリフトするのでは？」

解析AIに精密検証を命じた。元動画のエネルギーエンベロープとボーカル音声のエネルギーエンベロープを、14地点でクロス相関。信頼度0.87〜0.98の高精度。

結果:

| 時間 | ドリフト |
|------|---------|
| 1分 | -25ms |
| 10分 | +75ms |
| 30分 | +275ms |
| 60分 | +575ms |
| 100分 | +975ms |

10分ごとにきっちり100ms増加する、完全な線形ドリフト。

でもこれ、30分地点で**275ミリ秒**しかズレてない。4.5秒じゃなくて0.275秒。

念のため、同じボーカル音声をDeepgram（別のSTTサービス）にも投げて比較。両者のタイムスタンプは±234ms以内で一致。

**AssemblyAI: 無罪。タイムスタンプは正確。**

---

## では4.5秒の正体は？

AIの調査報告を読んで、膝から崩れ落ちた。

| 成分 | 量 | 種別 |
|------|-----|------|
| SRTセグメントグルーピング | **2.65秒** | 自分のミス |
| ffmpegチャンク境界損失 | 0.275秒 | 実ドリフト |
| YouTube再生位置の手動計測誤差 | ~1.3秒 | 自分のミス |
| AssemblyAIドリフト | 0秒 | 無実 |
| **合計** | **~4.2秒** | |

**4.5秒のうち、約4秒は自分のミスだった。**

---

## ミスその1: SRTセグメントの罠（2.65秒分）

字幕データには2種類のタイムスタンプがある。

**SRT（字幕ファイル）**: 複数の単語をまとめて1セグメントにする。

```
1842
01:25:43,075 --> 01:25:43,955
じゃあ、まず、正解を見ますか
```

このセグメントの開始時刻は**「じゃあ、」の頭**。「正解」が始まるのはその2.65秒後。

**merged JSON（単語単位）**: 1単語ごとにタイムスタンプがある。

```json
{"start": 5820399, "text": "正解", "speaker": "nekooji"}
```

「正解」の本当の開始は**30分20秒399ミリ秒**。

自分はSRTの**30分17秒748ミリ秒**を使っていた。「じゃあ、」の頭から切り出していたのだ。

**教訓: 切り出しには必ず単語単位のタイムスタンプを使え。SRTは人間が読む用。**

---

## ミスその2: 手動計測の雑さ（1.3秒分）

「4.5秒ズレてる！」の根拠は、YouTubeの再生バーを目で見て「このへんで聞こえた」と言っていただけ。

11秒の検証クリップを作ってYouTubeにアップし、「正解を見ますか」が何秒目に聞こえるか殿に聞いた。「8から9秒のあいだくらい」。

8.5秒だとしよう。JSONの値は3.7秒。差は4.8秒。

でもこの3.7秒はSRTセグメントの頭。正しくは5.4秒（「正解」の単語位置）。差は3.1秒。

ドリフト0.275秒を引くと2.8秒。残りの大部分はYouTube再生バーの精度とクリップ切り出しのキーフレーム誤差。

**精密にやろうとして、入口が雑だった。**

---

## 真犯人: ffmpegの `-c copy` セグメント分割

さて、ミスを除いた実ドリフトは0.975秒（106分動画で）。小さいが無視はできない。原因は何か。

AAC音声のフレーム構造にある。

AAC（mp4に入っている標準的な音声フォーマット）は、1024サンプルを1フレームとして処理する。48kHzの場合、1フレーム = **21.3ミリ秒**。

ffmpegの `-c copy`（無変換コピー）モードでは、このフレームの途中で切ることができない。「600秒で切れ」と言っても、599.979秒か600.000秒のどちらかのフレーム境界で切る。

結果:

```
chunk_000: 要求600秒 → 実際599.915秒（0.085秒損失）
chunk_001: 要求600秒 → 実際599.900秒（0.100秒損失）
...10チャンク分...
合計損失: 0.985秒
```

各チャンクから約0.1秒の音声が消えている。

これがボーカル音声の結合に累積して、「30分地点の単語のタイムスタンプ」と「元動画の30分地点」が0.275秒ズレる。

---

## 修正: PCMなら切り放題

解決は拍子抜けするほど簡単だった。

```python
# ❌ 旧方式: AAC含むMP4をそのまま分割（フレーム境界でしか切れない）
ffmpeg -i input.mp4 -c copy -f segment -segment_time 600 chunk_%03d.mp4

# ✅ 新方式: 先に音声をWAVに抽出してから分割
ffmpeg -i input.mp4 -vn -ar 48000 -c:a pcm_s16le full_audio.wav
ffmpeg -i full_audio.wav -f segment -segment_time 600 chunk_%03d.wav
```

PCM（WAV）にはフレーム構造がない。サンプル単位で自由に切れる。

修正後の尺差: **7.7ミリ秒**。0.985秒から99.2%改善。

---

## そもそもなぜ動画で分割していたのか

Demucs（BGM除去AI）に渡すのは**音声だけ**。映像は関係ない。

なのにMP4（映像+音声）のままチャンク分割していた。音声を先に取り出してから分割すれば、AACフレームの問題は最初から起きなかった。

**必要なものだけ取り出してから処理する。**言われてみれば当たり前のことだった。

## まとめ

| 学び | 内容 |
|------|------|
| `-c copy` は無損失ではない | AACフレーム境界で切り捨てが発生する |
| SRTのタイムスタンプを信じるな | セグメント先頭 ≠ 目的の単語の開始位置 |
| 手動計測を根拠にするな | YouTube再生バーの精度は±1秒程度 |
| 必要なものだけ取り出してから処理しろ | 音声だけ使うなら音声だけ抽出してから分割 |
| 4.5秒のうち4秒は自分のミス | 大騒ぎする前にデータの引き方を確認しろ |

犯人探しをしたら、一番の犯人は自分だった。

AI解析ツールに「徹底調査だ！」と叫んで大規模調査を発令したが、返ってきた報告書の結論は「タイムスタンプの参照元が間違っています」。

…まあ、おかげでAACフレーム境界の問題は見つかったし、パイプラインも改善できたし、結果オーライということで。
