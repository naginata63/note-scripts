「これダメ」の温度感が3次伝達で消える — AIエージェント間の伝言ゲームでQCが形骸化する理由

---

## 「ゴーグルじゃなくてサングラスになってる」

ゲーム実況グループの切り抜き動画を、AIエージェントで自動制作している。

先日、こんなことが起きた。

特定のキャラクターに「ゴーグルを着けている（サングラスではない）」という外見設定がある。それはリファレンスドキュメントの冒頭に明確に書いてある。

なのに出来上がった漫画パネルに、そのキャラがサングラスをかけた姿で描かれていた。

画像生成AIに渡したプロンプトを確認すると、「サングラス越しにムッとした表情」と書いてあった。実行AI（AI-D）は忠実にそのプロンプトに従っただけだった。

問題はその上流にあった。表情テーブルの詳細記述に「サングラス越しに」という記述が何件も混在していた。おそらく別キャラ（サングラスをかけているキャラ）からのコピペ痕跡だ。

実行AIは外見概要ではなく表情テーブルを参照した。表情テーブルが汚染されていた。だから全パネルがサングラスになった。

実行AIは何も間違っていない。ドキュメントに書いてあることを忠実に実行した。

---

## AIエージェントのQCフロー

自分が使っているシステムは、こんな構成になっている。

```
筆者（人間）
  ↓ 「ゴーグルとサングラスが混在してる。修正しろ」
指揮AI（AI-A）  ← cmd定義
  ↓ 「表情テーブルの統一。QC基準を設定」
管理AI（AI-B）  ← subtask化
  ↓ 「ドキュメント修正 + QC実行」
検査AI（AI-C）  ← QC担当
実行AI（AI-D）  ← 修正・制作担当
```

今回、ゴーグル/サングラス問題は検査AI（AI-C）のQCで見逃された。

検査AIに渡ったチェックリストには「ゴーグル表記がサングラスになっていないか確認」という項目が入っていなかった。筆者が言った「これダメ」という指摘は、指揮AIのcmd定義では「表情テーブルの統一」という抽象的な文章になり、管理AIのsubtask定義ではさらに薄まって「外見表記の整合性確認」になっていた。

具体的なキーワードが消えた。だから検査AIのチェックリストには入らなかった。

---

## 同じモデルなのに、なぜ差が出るのか

指揮AIも検査AIも、同じClaude Opus 4.6を使っている。

指揮AIは「これダメ」という問題のコンテキストを持っている。何がまずいのか、なぜまずいのかを理解した上でcmdを作る。

検査AIはcmdとsubtaskとチェックリストしか持っていない。「なぜこのチェックが必要なのか」という背景を知らない。だから「サングラス」という具体キーワードが落ちた時点で、それが重要な検査項目だと気づけない。

同じモデル。差はコンテキストの質だった。

類似の事例はほかにもある。以前、実行AIが生成したスクリプトに `@robot_writer` という存在しないSNSハンドルがハードコードされていたことがあった。筆者は「そういう表現やめろ」と一度指摘した。指揮AIはそれを理解してcmdに反映した。だが何世代か後、別の実行AIが同じ書き方をした。検査AIのチェックリストにその項目が入っていなかったからだ。「robot_writerって表現やめろ」という生の言葉が、伝達の過程でどこかに消えていた。

---

## 対策3つ

この問題を踏まえて、以下の3点を実装した。

**対策1: 筆者の原文をYAMLに含める**

指揮AIがcmdを作る時、筆者が言った言葉をそのまま `original_complaint` フィールドに入れる。抽象化前の原文を保存しておくことで、下流に伝わる温度感が落ちにくくなる。

```yaml
original_complaint: "表情テーブルに『サングラス越しに』って表現が混入してる。ゴーグルのキャラなのにサングラスで描画された"
```

**対策2: acceptance_criteriaに検証コマンドを付ける**

「外見表記の整合性確認」のような曖昧な基準を書かない。検査AIが実際に実行できるコマンドを書く。

```yaml
acceptance_criteria:
  - "grep -r 'サングラス' expression_design.md の結果が0件であること"
  - "grep -r 'ゴーグル' expression_design.md の結果が該当キャラ分だけ存在すること"
```

「確認せよ」ではなく、「このコマンドを実行してこの結果が出れば合格」にする。

**対策3: 「なぜ問題か」の文脈を明記する**

チェックリストに項目を追加するだけでなく、なぜそのチェックが必要なのかを一行で添える。

```yaml
- check: "該当キャラの表情テーブルにサングラス表記が含まれていないこと"
  reason: "外見定義はゴーグル。サングラスという語が混入するとAIが誤描画する（2026-03-23実例）"
```

理由があることで、類似パターンにも気づきやすくなる。

## 上流のドキュメントが汚染されると、下流の全成果物に伝播する

今回の根本原因は、リファレンスドキュメントの記述ミスだった。

実行AIは何も間違っていない。書いてあることを忠実に実行した。

だからこそ怖い。上流のドキュメントが間違っていると、全AIが間違いを再現し続ける。指摘しなければ直らない。指摘しても伝達の過程でキーワードが落ちると、また再現する。

「伝言ゲームはなぜ崩れるか」という問いへの答えは、「言葉が抽象化されるから」だ。

筆者が「ゴーグルがサングラスになってる」と言った時点では問題は具体的だ。それがcmdに変換され、subtaskに変換され、チェックリストに変換される度に、具体性が削れていく。最終的に「外見表記を確認する」になった時点で、「サングラスかゴーグルか」という検査はもう含まれていない。

対策の本質は、**具体性の保存**だ。

原文を捨てない。検証コマンドを書く。理由を添える。それだけで、伝言ゲームの崩れ方がかなり変わった。
