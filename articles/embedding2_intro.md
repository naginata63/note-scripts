# Gemini Embedding 2 — テキストも画像も動画も、ぜんぶ同じ「意味」で検索できるAIが出た

---

2026年3月10日、Googleが「Gemini Embedding 2」をPublic Previewとして公開した（GA未到達）。

ひとことで言うと、**テキスト・画像・動画・音声・PDFを全部同じベクトル空間に埋め込めるマルチモーダルEmbeddingモデル**だ。

これまでのEmbeddingモデルは基本的にテキスト専用だった。画像を扱いたければCLIP、音声ならWhisper→テキスト変換→Embedding、PDFならOCR→テキスト変換→Embedding。モダリティごとに別々のパイプラインが必要で、しかも変換を挟むたびに情報が落ちる。

Gemini Embedding 2は、**5つのモダリティを1つのモデルで直接ベクトル化**する。変換なし、パイプラインなし。写真1枚とテキスト1行が同じ3072次元のベクトルになり、コサイン類似度で比較できる。

## スペック詳細

| 項目 | 内容 |
|------|------|
| モデルID | `gemini-embedding-2-preview` |
| 入力モダリティ | テキスト / 画像 / 動画 / 音声 / PDF |
| テキスト入力上限 | 8,192トークン（前モデルtext-embedding-004は2,048トークン。4倍に拡大） |
| 画像入力 | 1リクエストあたり最大6枚（PNG / JPEG） |
| 動画入力 | 最大120秒（MP4 / MOV） |
| 音声入力 | 最大80秒（テキスト変換不要、ネイティブ音声理解） |
| PDF入力 | 最大6ページ（OCR内蔵。図表・レイアウトも理解） |
| 出力次元 | 3,072次元 |
| Matryoshka表現 | 768 / 1,536 / 3,072次元から選択可。精度とストレージのトレードオフを後から調整可能 |
| 対応言語 | 100言語以上 |
| 利用形態 | Gemini API（Google AI Studio）/ Vertex AI |
| ステータス | Public Preview（GA未到達、2026-03-10公開） |

### テキスト入力上限の拡大が地味に大きい

前モデル（text-embedding-004）は2,048トークンが上限だった。日本語で約1,500文字程度。長めの記事やドキュメントを埋め込もうとすると、チャンク分割が必須だった。

Gemini Embedding 2は8,192トークン。日本語で約6,000文字、A4用紙3-4枚分をそのまま1つのベクトルにできる。チャンク分割の手間が大幅に減る。

### Matryoshka表現（MRL）とは

ベクトルの次元数を後から選べる仕組み。3,072次元で保存しておけば、検索速度を上げたいときは先頭768次元だけ使えばいい。ベクトルを切り詰めるだけなので再計算不要。

精度への影響も軽微。MTEB（大規模テキストEmbeddingベンチマーク）のスコアは：
- 3,072次元: 68.16
- 1,536次元: 68.17（むしろ微増）
- 768次元: 67.99

768次元でもほぼ同等の精度が出る。ストレージは4分の1になる。

## 8つのタスクタイプ

Gemini Embedding 2の特徴的な機能として、**用途に応じてベクトルを最適化できるタスクタイプ指定**がある。同じテキストでも、検索用と分類用では最適なベクトル表現が異なる。タスクタイプを指定するだけで、モデルが内部的に最適化してくれる。

| タスクタイプ | 用途 | 使いどころ |
|-------------|------|-----------|
| RETRIEVAL_QUERY | 検索クエリ側 | ユーザーの検索文をベクトル化するとき |
| RETRIEVAL_DOCUMENT | 検索ドキュメント側 | 検索対象のドキュメントをインデックスに入れるとき |
| SEMANTIC_SIMILARITY | 意味的類似度 | 2つのテキスト・画像がどれだけ似ているか比較 |
| CLASSIFICATION | 分類 | テキストや画像をカテゴリに分類 |
| CLUSTERING | クラスタリング | 大量データをグループ分け |
| CODE_RETRIEVAL_QUERY | コード検索 | 自然言語でコードを検索（「エラーハンドリング」→該当コード） |
| QUESTION_ANSWERING | 質問応答 | FAQ検索、ナレッジベース問い合わせ |
| FACT_VERIFICATION | 事実検証 | 記述が既存データと矛盾していないかチェック |

特に**RETRIEVAL_QUERY / RETRIEVAL_DOCUMENT の非対称ペア**が実用的。検索インデックスを作る側と、ユーザーがクエリを投げる側で別々のタスクタイプを指定する。これだけで検索精度が上がる。

## こんなのに使えそう① — 領収書の写真を撮るだけで経費を自動仕分け

個人事業主やフリーランスの確定申告。1年分の領収書を科目ごとに分類するのは毎年の苦行だ。

Gemini Embedding 2なら、領収書の写真をそのまま埋め込んで**CLASSIFICATIONタスクタイプで経費科目ごとに自動分類**できそう。OCR内蔵だから、手書きの領収書でも金額や品目を読み取ってベクトル化する。

さらに、過去の領収書を全部インデックスに入れておけば「去年のモニターのレシート」と検索するだけで該当の写真が出てくる。テキストで検索して画像がヒットする、クロスモーダル検索の典型的なユースケース。

画像1枚の処理コストは約0.02円。300枚やっても6円。

## こんなのに使えそう② — 技術書PDF 50冊をまるごと横断検索

エンジニアなら誰しも覚えがあるだろう。「あのデザインパターンの図、どの本に載ってたっけ？」

PDFのテキスト検索では図表やコード例はヒットしない。Gemini Embedding 2なら、PDFを6ページずつチャンク分割してRETRIEVAL_DOCUMENTで埋め込めば、**テキストで検索して図表ページがヒットする**。CODE_RETRIEVAL_QUERYを使えば、自然言語でコードスニペットを探すこともできそう。

Matryoshka表現で768次元に落とせば、50冊（約1,500チャンク）でインデックスサイズは約4.5MB。ノートPCのメモリで余裕で動く。Python + FAISS + google-genai SDKの3点セットで、週末に作れる規模感。

## こんなのに使えそう③ — UIテストのスクショを「意味で」比較する

フロントエンドのVisual Regression Testing。ベースラインのスクショと新しいスクショをピクセル比較すると、フォントレンダリングの微差でfalse positiveが大量に出る。

Gemini Embedding 2のSEMANTIC_SIMILARITYなら、**2枚のスクショを「意味的に」比較**できそう。ピクセルが1px違っていても、レイアウトが同じなら高い類似度スコアが出る。逆に、レイアウトが崩れていれば低スコア。

ダークモード切り替えやフォントサイズ変更でもレイアウトが維持されていれば「同じ」と判定される。これは従来のピクセル比較では不可能だったこと。CIに組み込めば、false positiveの嵐から解放されるかもしれない。

## まとめ

Gemini Embedding 2は、テキスト・画像・動画・音声・PDFの5モダリティを1つのベクトル空間に統合する。この「統合」が本質で、別々のパイプラインを組まなくても異なるメディア間の検索・分類・比較ができるようになる。

まだPublic Preview（モデルID: `gemini-embedding-2-preview`、GAは未到達）なので本番利用は慎重に。でも無料枠で試せる。触ってみる価値はある。
