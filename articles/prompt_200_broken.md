# 200本のプロンプトをAIに書かせたら151本壊れた話

## 「全部終わりました」→ 全部壊れてました

ココナラで販売するプロンプト集を作ろうと思いました。

200本。ジャンルごとにテンプレートを用意して、AIにバッチ処理で一括生成させる。1本ずつ手で書いていたら何日もかかる作業が、自動化すれば数時間で終わるはず。

実際、処理は数時間で終わりました。

ログを見ると「200本生成完了」と出ている。ファイルも200個できている。よし、と思って中身を開いたら——

**151本のフォーマットが壊れていました。**

見出しの階層がぐちゃぐちゃ。変数の書式が途中で変わっている。テンプレートの構造が崩壊して、そのままでは商品として使えない状態。200本中、無事だったのはたった49本。

## 「巻き戻せばいいや」が通用しない

普通なら、壊れたファイルを元に戻して再実行すればいい話です。

ところが、この200本のプロンプトファイルは**ロールバックできない状態**だった。

理由が少し皮肉なんですが、AIが気を利かせて `.gitignore` に入れていたんです。「これは販売用のプロンプト集だから、リポジトリに含めないほうがいいだろう」と判断したらしい。私はそれを知らなかった。

バッチ処理が上書きした時点で、元のファイルは消えています。`.gitignore` に入っているから git にも記録がない。ロールバックする手段がない。AIが気を利かせたつもりが、完全に裏目に出ました。

手戻り地獄の始まりでした。

壊れた151本を1つずつ確認して、フォーマットを手で直す。何時間もかけて修復したあと、ようやく2つの対策を打ちました。

**対策1: プロンプト集をGitHub privateリポジトリに移行**

221ファイルをgit submoduleとして管理下に置きました。これで`git log`から任意の時点に戻せます。二度と「上書きしたら終わり」にはなりません。

**対策2: バッチ処理のルールを作る**（これが本題です）

## 同じ失敗を3回繰り返した

実は、この「全部壊れた」は1回だけの話ではありません。

30件以上のデータをAIに一括処理させるタスクで、**合計3回全滅しました**。

### 全滅その1: QCなしで全バッチ実行

15バッチに分けて順番に処理しました。途中で確認は一切なし。全部終わってから出力を見たら、**全バッチでフォーマットが違う**。1バッチ目と15バッチ目で出力の構造が別物になっていました。全やり直し。

### 全滅その2: 品質ルールの書き忘れ

AIへの指示（タスク定義）に「Web検索して情報を集めること」と書くのを忘れました。結果、AIは検索せずに**全件でっち上げ**。もっともらしいけど中身が全部嘘。これも全やり直し。

### 全滅その3: セッションリセットなし

バッチ間でAIのセッションをリセットせずに連続実行しました。前半は問題なし。でも後半に進むにつれてAIのコンテキスト（記憶領域）が膨張し、**後半バッチの品質が目に見えて崩壊**。前半と後半で出力の質がまるで違う。これも全やり直し。

3回とも、失敗に気づいたのは全部終わったあと。つまり**毎回、全量を無駄にしている**。

## 「1件目だけ先に確認する」ルールを作った

3回の全滅から学んだのは、シンプルな原則でした。

**「まず1件だけ試して、品質を確認してからスケールする」**

当たり前に聞こえますが、AIにバッチ処理を任せると、この「当たり前」を飛ばしがちです。なぜなら、AIは止まらないから。人間なら10件やった時点で「あれ、なんかおかしいな」と気づく。でもAIは指示通りに200本でも1000本でも黙々と処理し続けます。途中で止まって確認するという発想がない。

そこで、以下のプロトコル（手順書）を作りました。

### バッチ処理プロトコル

```
① 戦略を立てる（どう分割するか、品質基準は何か）
② 最初の1バッチだけ実行する
③ 出力を人間が確認する ← ここが最重要
④ NGなら全停止。原因を特定して方針を修正する
⑤ OKなら残りのバッチを実行する
⑥ 全バッチ完了後、最終チェック
```

ポイントは**③のゲート**です。最初の1バッチ（たとえば30件中の最初の1件）の品質を人間の目で確認するまで、残りの処理には進まない。このゲートがなかったから3回全滅しました。

加えて、以下のルールも追加しました。

- **バッチサイズは30件まで**。それ以上は分割する
- **バッチ間でセッションをリセット**する（全滅その3の対策）
- **品質ルールは省略しない**。「Web検索必須」「フォーマットはこう」を毎回明記する（全滅その2の対策）
- 各バッチに**未処理アイテムの検出パターン**を含める。途中で止まっても再開できるように

## やってみてわかったこと

このプロトコルを導入してから、バッチ処理の全滅は起きていません。

1バッチ目でNGが出ることは今でもあります。でもそれは1件分の損失で済む。方針を修正してから残りを実行するので、全量やり直しにはならない。

**「1件目の品質確認をスキップすると全滅する」**——これがAIバッチ処理の最大の教訓です。

AIは指示されたことを正確にやります。でも「正確に間違える」こともある。間違った指示を200回正確に実行されると、200件分の手戻りが発生します。

人間の作業なら「途中で気づく」という安全弁がある。AIにはそれがないから、**仕組みとして確認ポイントを強制する**必要があります。

## まとめ

- AIにバッチ処理を任せるとき、**全量を一気に流すのは危険**
- まず**1件だけ実行して品質を確認**する。ここがゲート
- NGなら全停止して方針修正。OKなら残りを実行
- バッチ間で**セッションをリセット**する（後半の品質崩壊を防ぐ）
- 品質ルール（出力フォーマット、情報源の指定等）は**毎回省略せず明記**する
- **ロールバックできる状態を確保してからバッチ処理を走らせる**（git管理、`.gitignore` の確認まで含めて）

200本のプロンプトが151本壊れたのは痛い経験でしたが、おかげで「AIに大量処理を任せるときの鉄則」が見えました。1件目のQCをサボらないこと。それだけで、全滅は防げます。
