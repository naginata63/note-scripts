# 設定ファイルを作ったのに誰も読んでいなかった話

![カバー画像](../images/dictionary_dead_cover.png)
*設定ファイルは書くだけでは動かない*

## 「辞書あるよ？」「え？」

AIを使った開発を続けていると、同じ間違いが何度も起きることがあります。

表記揺れ、用語の不統一、誤変換。そのたびに手動で直す。「これ前も直したな……」と思いながら。

あるとき、この問題を根本的に解決しようと思い立ちました。

**辞書ファイルを作ろう。**

正しい用語と、よくある間違いのペアを一覧にして、YAMLファイルに書き出しました。ファイルの冒頭には丁寧にコメントまで書きました。「このファイルはLLMに渡して表記を修正させるために使う」と。

そして数週間が経ち、ふと気づきました。

**どのスクリプトにもこのファイルを読み込むコードがない。**

辞書ファイルは作った。存在もしている。でも**どのプログラムにもインポートされていない**。つまり完全に死んでいた。

## 同じパターン、もう一つ

調べてみると、同じことがもう一箇所で起きていました。

プロジェクトには「マスターデータ」と呼べる設定ファイルがありました。正式な名称や属性が定義されていて、UIの表示部分ではちゃんと使われている。ところが**データ生成のパイプライン**には一切渡されていなかったのです。

結果、AIが生成するテキストには毎回カタカナ表記や独自の言い回しが混ざり、人間が手動で直していました。

正しいデータはそこにあるのに、使われていない。

## なぜ起きるのか

これは「作って満足」問題です。開発あるあるとも言えます。

1. **設定ファイルを作る** → 達成感がある
2. **コメントに「ここで使う」と書く** → さらに達成感
3. **実際にインポートするコードを書く** → 忘れる

特にAI開発では、この3番目のステップが抜けやすい。なぜなら**AIに「辞書を作って」と指示すると、ファイルの作成までは完璧にやる**からです。でも「それを既存のパイプラインに組み込む」という横断的な作業は、別の指示として明示しないと実行されません。

人間のチーム開発でも似たことは起きます。「設計書は書いたけどコードに反映されていない」「環境変数を.envに追加したけどアプリが読んでいない」。ただ、AIの場合はレビュー工程がないぶん、気づくまでの時間が長くなりがちです。

## 対策: 3点セットで考える

この経験から、設定ファイルを作るときは**3点セット**で考えるようにしました。

**1. ファイルを作る**（当然）

**2. 読み込むコードを書く**（ここが抜ける）

設定ファイルを作ったら、同じタスクの中で「これを読み込んで使うコード」まで書く。別タスクに分けると忘れます。

**3. 使われていることを検証する**（仕上げ）

grepで「そのファイル名がインポートされているか」を確認するだけでも十分です。

```
grep -r "custom_vocabulary" scripts/
```

これで0件なら、ファイルは死んでいます。

## もう一つの対策: 出力側にも防壁を

入力側（プロンプトに辞書を渡す）だけでなく、**出力側にも後処理を入れる**のが堅実です。

AIの出力テキストに対して、辞書の「誤表記→正表記」マッピングで単純な文字列置換をかける。これなら、たとえプロンプトでの指示が効かなくても、最終出力は正しい表記になります。

入力で教えて、出力で検品する。二重の防壁です。

## まとめ

設定ファイルは**書いただけでは動かない**。当たり前のことですが、AIを使った開発では特にこの罠にはまりやすい。

- 辞書・マスターデータ・設定ファイルを作ったら、**同じタスク内で組み込みまでやる**
- 定期的に `grep` で参照箇所を確認する
- 出力側の後処理も併用して二重防壁にする

「ちゃんと作ったのに使われていない」。これに気づけるかどうかが、AI開発の品質を分けるポイントだと思います。
