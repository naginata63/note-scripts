## 🤖 「この記事、AIが書いてAIが投稿しました」

「書くネタはあるのに、投稿が面倒で後回しになる」

これ、あるあるじゃないでしょうか。Markdownで下書きを書いて、noteのエディタに貼り付けて、画像を設定して、タグを付けて……とやっていると、「書く」より「投稿する」ほうが手間だと感じることがあります。

であれば、投稿作業をAIに丸投げできたら最高では？　そう思って作ったのが、今回紹介するnote自動投稿ツールです。そしてこの記事自体も、そのツールを使って投稿しています。

## やったこと

GitHubでnote-scriptsというOSSリポジトリを公開しています。

コアとなるのはnote-edit.jsというスクリプト。コマンド1行で、ローカルのMarkdownファイルをnoteの下書きとして投稿できます。

```
node scripts/note-edit.js --action=create --title="タイトル" --body-file=article.md
```

仕組みはPlaywrightを使ったブラウザ自動操作です。実際にChromeを起動し、人間がやるのと同じようにnoteのエディタを操作して投稿します。Markdownは自動的にリッチテキストに変換され、h2/h3見出し・太字・リンク・引用・箇条書き・コードブロックに対応しています。

認証はNOTE_EMAILとNOTE_PASSWORD環境変数で設定するだけ。一度ログインするとセッションが保存されるので、2回目以降はスムーズです。

既存の下書きを更新したい場合は`--action=edit`オプションでnoteの記事URLを指定します。

```
node scripts/note-edit.js --action=edit \
  --url="https://note.com/n/xxxx" \
  --body-file=article.md
```

また、画像の追加も1コマンドで可能です。`--image-file`オプションで記事本文に画像を挿入でき、`--cover-image`オプションでカバー画像（アイキャッチ）を設定できます。AIが生成した画像を自動で付けて投稿するところまでを一気通貫で実現しています。

## 苦労した点 🔧

実はここまでの道のりが長かったです。

**最初の挑戦：非公式API**　noteには公開APIがないため、ブラウザの開発者ツールで内部APIを調べて直接叩く方式を試みました。最初はうまくいったのですが、仕様変更でたびたび壊れるため断念。「APIは信頼できない」という結論に至りました。

**第2の挑戦：ヘッドレスモード**　Playwrightのヘッドレスモード（ブラウザ画面なし）で試したところ、reCAPTCHAに阻まれてログインできませんでした。ボットと判定されるのか、自動化検知に引っかかるのか、なかなか突破できず。

**解決策：headedモード**　ブラウザ画面を実際に表示する「headedモード」に切り替えたところ、あっさり動きました。グラフィカルな環境（DISPLAYあり）が必要になりますが、Linux環境でDISPLAY=:0を設定すれば問題ありません。

**エディタの仕組み**　noteのエディタはTiptapベースのProseMirrorを使っています。ProseMirrorにはMarkdownの「input rule」機能があり、##と入力すると自動的にh2見出しに変換される仕組みがあります。これを利用してMarkdownをリッチテキストとして入力することで、確実な変換を実現しています。

**画像生成モデルの選定と最適化**　挿絵の自動生成にも試行錯誤がありました。最初はGeminiのフラッシュモデルで試していましたが、より高品質な画像を求めてGoogle DeepMindのImagen 4（imagen-4.0-generate-001）に切り替えました。さらにその後、テキストと画像を統合的に扱えるNano Banana 2（gemini-3.1-flash-image-preview）に移行しました。このモデルは既存の画像を参照しながら新しいバリエーションを生成できるため、記事シリーズ全体でビジュアルの統一感を保ちやすい点が優れています。テキストを日本語で直接画像に描き込める点も便利で、記事の内容に合ったオリジナル画像を自動生成できるようになっています。

**ログイン判定の改善**　セッション管理にも改善を重ねました。Playwrightの`launchPersistentContext`でセッションを保存していますが、長時間経過するとセッションが切れることがあります。当初はURLのパスでログイン状態を判定していましたが、noteがリダイレクト前に一時的にログインページのURLを返すことがあり、誤判定が起きていました。そこで`ensureLoggedIn()`関数にリダイレクト検知方式を採用し、最終的に落ち着いたURLとDOM状態の両方を確認する仕組みにしました。これにより、セッション切れによる突然の失敗がほぼなくなりました。

**Markdown変換の安定化**　見出し記法（## や ###）の変換でもハマりポイントがありました。ProseMirrorのinput ruleは「空の段落ブロックの先頭で ## と入力」したときに発火するのですが、入力タイミングが少しでもずれると変換されず記号がそのまま表示されてしまいます。wait時間を調整してinput ruleの発火を確実にすることで、この問題を解決しました。また太字の変換方式を`**bold**`のinput ruleからCtrl+Bショートカットへ切り替えることで、箇条書き内で太字を使ってもリスト構造が崩れない安定した実装を実現しています。コードブロックの脱出処理もCtrl+Enter方式に改めており、全体的なMarkdown変換の精度が大幅に向上しています。

## 使い方

シンプルです。

**基本コマンド（新規下書き作成）:**

```
node scripts/note-edit.js --action=create \
  --title="記事タイトル" \
  --body-file=path/to/article.md
```

**対応するMarkdown記法:**

- ## / ### で見出し（h2/h3）
- **テキスト** で太字
- 引用ブロック・箇条書き・コードブロック

**画像付きで投稿する場合:**

```
node scripts/note-edit.js --action=edit \
  --url="https://note.com/n/xxxx" \
  --image-file=path/to/image.png
```

`--image-file`を指定すると記事の末尾に画像が追加されます。カバー画像（アイキャッチ）を設定したい場合は`--cover-image`オプションを使います。複数の画像を別々のコマンドで追加することもできます。

詳しくはGitHubリポジトリのREADMEを参照してください。

## 技術的な詳細

**Playwrightによるブラウザ制御**

このツールの心臓部はPlaywrightです。Playwrightは、Microsoft製のブラウザ自動化ライブラリで、Chromium・Firefox・WebKitに対応しています。note-edit.jsでは、Chromiumを使って実際のブラウザウィンドウを起動し、noteのエディタを操作します。

headlessモードではなくheadedモードを採用した理由は「reCAPTCHA対策」です。ヘッドレス環境ではGoogleのbotチェックに引っかかることが多く、ログインに失敗するケースが頻発しました。headedモードで実際の画面を表示することで、この問題を回避しています。Xサーバーが動いているLinux環境（WSL2 + VcXsrvなど）や、VNCなどのリモートデスクトップ環境があれば問題なく動作します。

**ProseMirrorエディタの操作**

noteのエディタはTiptap v2ベースのリッチテキストエディタです。内部的にはProseMirrorというフレームワークを使っており、JavaScriptのAPIでコンテンツを直接操作することも理論上は可能です。ただし、APIの内部仕様が変わると一気に動かなくなるリスクがあります。

そのため、note-edit.jsでは「Markdownのinput rule」を活用しています。ProseMirrorには「##と入力するとh2見出しに自動変換」「\*\*テキスト\*\*と入力すると太字に変換」などのinput rule機能があります。このルールを利用して、Markdownテキストをそのままエディタに貼り付けることで、確実なリッチテキスト変換を実現しています。

**画像アップロードの仕組み**

画像アップロードは少し複雑です。noteのエディタには「+ボタン」→「画像を追加」というUIがあり、ファイル選択ダイアログが開きます。Playwrightはこのファイル選択ダイアログを`page.waitForEvent('filechooser')`でインターセプトし、ローカルの画像ファイルパスを渡すことで画像をアップロードします。

カバー画像の設定は、記事の編集画面にある「カバー画像を設定」ボタンから別途行います。こちらもファイル選択ダイアログを通じてアップロードし、設定後に「保存」ボタンを押すという2ステップの処理が必要です。これらのUIセレクターはnoteのHTMLを解析して特定し、Playwrightのロケーターで操作しています。

**セッション管理とログイン維持**

Playwrightの`launchPersistentContext`を使ってプロファイルディレクトリに認証情報を保存しています。一度ログインするとCookieやlocalStorageが`~/.cache/ms-playwright/note-cli-profile`に保存されるため、以降はログイン処理をスキップできます。

## まとめと今後

このツールのポイントは「API依存ではなく、ブラウザGUIを自動操作する」という点です。APIが突然変わっても、人間が操作できる限りは動き続けます。

現在はAIエージェントシステム（多段階のClaude）と連携させていて、「テーマを渡すとAIが記事を書き、自動でnoteに下書きを投稿する」フローが動いています。ちょうどこの記事がそのデモです。

具体的には、複数のClaudeエージェントが役割分担して動いています。記事の企画・執筆を担当するエージェント、画像生成AIと連携して挿絵を作るエージェント、そしてnoteへの投稿を実行するエージェント——それぞれが非同期で協調し、最終的に「完成した記事が自動でnoteに投稿される」状態を実現しています。アイデア出しから公開まで、人間がほぼ介入しないワークフローを構成できるのが最大の魅力です。

今後は定期的な自動投稿スケジューリングや、SNSへの同時投稿なども視野に入れています。
